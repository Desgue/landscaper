### SRP Violations

**`canvas-pixi/CanvasHost.tsx` ‚Äî `initApp` async function:** ENG-47 extracted pan (`useCanvasPan`), keyboard shortcuts (`useCanvasKeyboardShortcuts`), zoom animator (`createZoomAnimator`), scene graph construction (`buildCanvasSceneGraph`), and renderer creation (`createAllRenderers`) into separate modules ‚Äî a meaningful improvement. The single `useEffect`/`initApp` body still owns: PixiJS app initialization, design token building, viewport store subscription wiring, pointer event registration (pan, cursor tracking, double-click), WebGL context loss/restore handlers, texture GC interval, texture atlas creation, tool handler construction (8 handlers inline), interaction manager setup, render scheduler start, and cleanup orchestration. The function still runs 300+ lines and remains a coordination surface for unrelated concerns.

**`canvas-pixi/InteractionManager.ts:67‚Äì436`:** `createInteractionManager` handles tool dispatch routing, boundary vertex/arc hit-testing (including sagitta math), SSM event packaging, direct store subscription wiring, and SSM lifecycle management. The hit-test geometry at lines 134‚Äì180 belongs in a dedicated boundary-hit module.

**`canvas-pixi/PlacementHandlers.ts` (entire file):** Five handler factories in one file. Each factory mixes collision checks, snap calls, project mutations, history management, and inspector selection. No shared interface or composition keeps them together beyond file size.

**`canvas-pixi/SelectionStateMachine.ts`:** `handleDown` is 190 lines and handles path-point drag initiation, multi-element resize handle detection, single-element resize, rotation, group selection resolution, and box-select start. These are all distinct sub-operations, none depending on the other.

**`canvas-pixi/useCanvasKeyboardShortcuts.ts`:** One hook covers undo, redo, snap toggle, grid toggle, delete, copy, paste, tab cycle, escape, group, and ungroup. Each shortcut is a self-contained action with separate store dependencies.

### Dependency Inversion

**`canvas-pixi/InteractionManager.ts:14‚Äì22`:** Imports `useProjectStore`, `useSelectionStore`, `useInspectorStore`, `useViewportStore`, and `useToolStore` at the top level. The canvas event layer has compile-time imports on every major domain store.

**`canvas-pixi/SelectionStateMachine.ts:13‚Äì18`:** Documented as "framework-agnostic" and "no Konva or PixiJS imports" but directly calls `useProjectStore.getState()`, `useSelectionStore.getState()`, `useHistoryStore.getState()`, `useViewportStore.getState()`, and `useToolStore.getState()` throughout. It cannot be instantiated or tested outside the store environment.

**`canvas-pixi/TerrainPaintHandler.ts:143‚Äì147`:** `paintCell` calls `useProjectStore.getState().registries` mid-algorithm to check blocked structure categories. A pure paint function reaching into the registry via a global store is a direct coupling.

**`canvas-pixi/PlacementHandlers.ts:316‚Äì352`:** Plant placement commits by calling `updateProject`, `pushHistory`, `markDirty`, and `setInspectedElementId` ‚Äî four different store APIs ‚Äî from inside a single `onPointerDown` handler with no intermediary.

**`store/useHistoryStore.ts:37‚Äì46`:** `undo` calls `useProjectStore.getState().loadProject()` and `markDirty()` directly. The history store has a runtime dependency on the project store with no interface boundary between them.

### Layer Coupling

**`canvas-pixi/InteractionManager.ts:278‚Äì284`:** `onPointerMove` queries `useProjectStore.getState().currentProject` for hover hit-testing on every pointer move event. The rendering event layer queries the domain store at pointer frequency.

**`canvas-pixi/useCanvasKeyboardShortcuts.ts:24‚Äì56`:** The canvas keyboard hook directly mutates `useViewportStore`, `useSelectionStore`, `useProjectStore`, and `useHistoryStore`. Canvas input handling reaches across all domain layers.

**`canvas-pixi/CanvasHost.tsx:305`:** `useBoundaryUIStore.getState().setBoundaryHandle(boundaryHandler)` pushes an imperative handle into a Zustand store so HTML overlays can call it. The PixiJS imperative object is exposed through a store that React components subscribe to.

**`canvas-pixi/BoundaryRenderer.ts:30‚Äì31`:** `BOUNDARY_COLOR` and `LABEL_COLOR` are module-level `let` variables mutated by `setTokens()`. Module-level mutable state is shared across all instances if two canvases ever exist.

### Improved Architecture

The four problem clusters above ‚Äî ceremony duplication, SRP violations in `initApp` and `handleDown`, direct store imports in every handler, and inspector callback cloning ‚Äî share a single root cause: **no intermediate layer between the canvas event code and the store layer**. Handlers reach into stores directly, so every mutation site reinvents the same orchestration and every handler is hard-wired to a specific set of store APIs.

The architecture change is one new layer and three targeted extractions.

---

#### A. `commitProjectUpdate` ‚Äî eliminate the 15-site ceremony

**Current** (repeated verbatim at 15+ sites):
```
const snapshot = structuredClone(proj)
updateProject(draft => { ‚Ä¶ })
pushHistory(snapshot)
markDirty()
```

**Proposed** ‚Äî single function in `store/projectActions.ts`:
```ts
export function commitProjectUpdate(
  actionName: string,
  updater: (draft: Project) => void,
): void {
  const store = useProjectStore.getState()
  const proj = store.currentProject
  if (!proj) return
  const snapshot = structuredClone(proj)
  store.updateProject(updater)
  useHistoryStore.getState().pushHistory(snapshot)
  store.markDirty()
}
```

Every call site collapses to one line. `actionName` feeds into the logging gap identified in Section 3 at zero extra cost.

**Note on `BoundaryHandler.doCommitBoundary`:** The analysis found a variation ‚Äî history is pushed *before* `updateProject`. This is the only site where the order differs and may be a latent bug: the snapshot is taken from `currentProject` and pushed before the project is updated, meaning the "previous" state in history is the state *at the time of push*, not before the mutation begins. All other sites snapshot then mutate. This ordering should be verified and aligned.

---

#### B. `CanvasContext` interface ‚Äî decouple handlers from stores

**Current dependency graph** (each arrow = direct store import):
```
PlacementHandlers  ‚Üí  useProjectStore, useHistoryStore, useViewportStore,
                       useInspectorStore, useStructureToolStore, usePlantToolStore,
                       useLabelToolStore, useMeasurementStore
SelectionStateMachine ‚Üí useProjectStore, useHistoryStore, useSelectionStore,
                        useViewportStore, useToolStore
BoundaryHandler    ‚Üí  useProjectStore, useHistoryStore, useViewportStore,
                       useToolStore, useBoundaryUIStore
PathDrawingHandler ‚Üí  useProjectStore, useHistoryStore, useViewportStore,
                       useToolStore, usePathToolStore
TerrainPaintHandler‚Üí  useProjectStore, useHistoryStore, useToolStore, useSelectionStore
InteractionManager ‚Üí  useProjectStore, useSelectionStore, useInspectorStore,
                       useViewportStore, useToolStore
```

Every handler has 5‚Äì8 direct store imports. A store API change ripples across all of them.

**Proposed** ‚Äî one `CanvasContext` interface passed in at construction time:
```ts
// canvas-pixi/CanvasContext.ts
export interface CanvasContext {
  // Read
  getProject(): Project | null
  getRegistries(): Registries
  getZoom(): number
  getActiveTool(): ToolType
  // Mutations
  commitUpdate(actionName: string, updater: (draft: Project) => void): void
  pushLiveUpdate(updater: (draft: Project) => void): void  // no history, for drag moves
  // Selection
  select(ids: string[]): void
  deselectAll(): void
  setInspectedElement(id: string | null): void
}
```

`CanvasHost` builds this object once (it already imports all stores) and passes it to every handler. Handlers drop their individual store imports entirely.

```
CanvasHost  ‚Üí  useProjectStore, useHistoryStore, useSelectionStore,
               useViewportStore, useToolStore   (imports stay here, not in handlers)
   ‚îÇ
   ‚îî‚îÄ CanvasContext instance
        ‚îÇ
        ‚îú‚îÄ‚îÄ PlacementHandlers(ctx)
        ‚îú‚îÄ‚îÄ SelectionStateMachine(ctx)
        ‚îú‚îÄ‚îÄ BoundaryHandler(ctx)
        ‚îú‚îÄ‚îÄ PathDrawingHandler(ctx)
        ‚îî‚îÄ‚îÄ TerrainPaintHandler(ctx)
```

Each handler now has **one** import (`CanvasContext`) and is testable by constructing a mock context object.

#### C. Shared `snapWorldPoint` ‚Äî consolidate three copies

**Current:** Three functions with identical bodies in `PlacementHandlers.ts:90`, `BoundaryHandler.ts:88`, `PathDrawingHandler.ts:45`. Only difference: `PlacementHandlers` parametrizes the snap context (`'place' | 'label' | 'measurement'`); the others hardcode `'place'`.

**Proposed** ‚Äî export from `snap/snapSystem.ts`:
```ts
export function snapWorldPoint(
  worldX: number,
  worldY: number,
  context: SnapContext,
  altKey: boolean,
  ctx: Pick<CanvasContext, 'getProject' | 'getZoom'>,
): { x: number; y: number }
```

With `CanvasContext` available in handlers, this function takes a narrow slice of it rather than calling stores itself. All three handler copies are deleted.

#### D. `useInspectorEdit<T>` ‚Äî eliminate per-inspector callback cloning

**Current:** `startEdit`, `commitEdit`, `update`, `updateLive` are copy-pasted in all 6 inspector files. The only variation is the type parameter and type guard (`el.type === 'plant'`, `el.type === 'structure'`, etc.).

**Proposed** ‚Äî single hook in `components/inspector/useInspectorEdit.ts`:
```ts
export function useInspectorEdit<T extends CanvasElement>(
  elementId: string,
  elementType: T['type'],
) {
  const updateProject = useProjectStore(s => s.updateProject)
  const pushHistory = useHistoryStore(s => s.pushHistory)
  const snapshotRef = useRef<Project | null>(null)

  const startEdit = useCallback(() => { ‚Ä¶ }, [])
  const commitEdit = useCallback(() => { ‚Ä¶ }, [pushHistory])
  const update = useCallback((updater: (el: T) => void) => { ‚Ä¶ }, [elementId, ‚Ä¶])
  const updateLive = useCallback((updater: (el: T) => void) => { ‚Ä¶ }, [elementId, ‚Ä¶])

  return { startEdit, commitEdit, update, updateLive }
}
```

Each inspector becomes:
```ts
const { startEdit, commitEdit, update, updateLive } = useInspectorEdit<PlantElement>(
  element.id, 'plant'
)
```

~40 lines of callbacks per file collapse to one line. The hook lives in `inspectorShared.tsx` or its own file.

#### E. `SelectionStateMachine.handleDown` ‚Äî split into mode initializers

**Current:** 190-line function dispatches into 6 drag modes: `path_point_dragging`, `resizing` (multi), `resizing` (single), `rotating`, `moving`, `box_selecting`. All share the same snapshot and store read boilerplate at their top.

**Proposed** ‚Äî dispatcher + 6 focused initializers:
```ts
// Each returns a drag state or null (no-op)
function initPathPointDrag(down: DownEvent, ctx: CanvasContext): DragState | null
function initResizeMultiDrag(down: DownEvent, ctx: CanvasContext): DragState | null
function initResizeSingleDrag(down: DownEvent, ctx: CanvasContext): DragState | null
function initRotateDrag(down: DownEvent, ctx: CanvasContext): DragState | null
function initMoveDrag(down: DownEvent, ctx: CanvasContext): DragState | null
function initBoxSelect(down: DownEvent, ctx: CanvasContext): DragState | null

function handleDown(e: DownEvent): void {
  drag =
    initPathPointDrag(e, ctx) ??
    initResizeMultiDrag(e, ctx) ??
    initResizeSingleDrag(e, ctx) ??
    initRotateDrag(e, ctx) ??
    initMoveDrag(e, ctx) ??
    initBoxSelect(e, ctx) ??
    null
}
```

Each initializer is ~25‚Äì35 lines, testable independently, and named by what it does.

## 2. Patterns and Architecture

### Misapplied or Poor Fit

| File | Current Pattern | Issue | Suggested Pattern |
|---|---|---|---|
| `store/useHistoryStore.ts:37‚Äì64` | Direct store-to-store coupling | `undo`/`redo` calls `useProjectStore.getState().loadProject()` directly; history store can't be tested without project store. | Mediator or Command Bus: undo/redo emit events or call a passed-in callback rather than importing a peer store. |
| `store/useProjectStore.ts:49‚Äì59` | Timer in Zustand state | `_saveTimer` lives in public store state, triggers subscriptions on every `markDirty()` call, and exposes a mutable timer handle to any subscriber. | Move the debounce timer into a module-level closure outside the store. No subscriber needs this state. |
| `store/useGenerateStore.ts:87‚Äì90, 244‚Äì265` | Module-level mutable globals + stub `setTimeout` | `activeController`, `timeoutTimer`, and `isTimeoutAbort` are module globals. `sendChatMessage` uses a hardcoded 2-second fake response. | API lifecycle state belongs in the store; stubbed features should use a strategy/adapter interface, not hardcoded `setTimeout` delays embedded in the production store. |
| `canvas-pixi/SelectionOverlay.ts:161‚Äì165` | Four parallel `connectStore` subscriptions | Four independent subscriptions each call `redraw()` synchronously on any store change. A single project-elements change triggers four successive redraws per frame. | Batch into one subscriber with a dirty flag; call `scheduler.markDirty()` and let the rAF loop coalesce. |
| `canvas-pixi/InteractionManager.ts:349‚Äì370` | Tool-change subscription with boundary teardown side-effect | The tool-change subscription cancels in-progress boundary drags and calls `boundary.destroy()` as a side-effect. A store subscription doing imperative teardown is unexpected and hard to trace. | Chain teardown into the tool store action or expose a reset method that CanvasHost calls when tool changes, not a hidden subscription consequence. |

### Over-engineered

**`components/inspectorSlots.ts` + `registerInspectorSlots.ts`:** A module-level `Map`, a React context with a default value pointing to the same `Map`, and a separate `registerInspectorSlots.ts` side-effect module whose only job is to call `registerInspectorSection` three times for three hard-coded components. `InspectorPanel` could just import those three components directly. The plugin seam adds two files and one load-order dependency with no actual dynamism.

**`canvas-pixi/DisposalManager.ts`:** Manually tracks container references for cleanup. PixiJS containers already cascade destroy to children. The only real work `DisposalManager` does is `registerContainer(world)` and `destroyAll()` ‚Äî which calls `world.destroy()`. That's one line in `CanvasHost`'s cleanup function.

**`canvas-pixi/connectStore.ts`:** Nine lines wrapping `store.subscribe` with a selector and a prev-value comparison. Zustand's subscribe API accepts a selector and equality function natively. The wrapper adds a file and import with no capability beyond what the library provides.

### Missing Patterns

**Command pattern for mutations.** Every action site calls `structuredClone(project)`, then `updateProject(...)`, then `pushHistory(snapshot)`, then `markDirty()`. That 4-step ceremony appears 15+ times. A single `commitProjectUpdate(actionName, updater)` function in the store layer would be the integration point for both history and dirty-flagging, and would make every mutation site readable in one line.

**Shared inspector edit hook.** The `startEdit`/`commitEdit`/`update`/`updateLive` pattern in every per-type inspector should be a `useInspectorEdit<T>(elementId)` hook that returns these four callbacks generically. The ENG-64 split created the right file structure for this but left the duplication in place.

**Coordinate type safety.** Screen coordinates, world coordinates, and canvas-relative coordinates flow through `CanvasHost`, `viewport.ts`, `InteractionManager`, and all tool handlers as bare `{x, y}` or raw number pairs. Branded types (`ScreenPoint`, `WorldPoint`) would make coordinate-space bugs compile errors rather than runtime surprises during interaction regressions.

**Object pool for PixiJS display objects.** Renderers create and destroy `Sprite`, `Text`, and `Graphics` objects as element counts change. PixiJS GC pressure from rapid create/destroy is documented (referenced in `BaseRenderer.ts:93` for `Graphics`). The same reasoning applies to sprites in `PlantRenderer` and `StructureRenderer`.

**Adapter for store access in canvas layer.** All five tool handlers and the SSM call `getState()` on four to six stores. Passing a `CanvasContext` interface containing only the methods those handlers need would make their dependencies explicit, testable, and refactorable without touching every handler when a store API changes.

## 3. Logging and Debuggability

No structured logger exists. The codebase has ad-hoc `console.debug/warn/error` at roughly 30 call sites, with another 30+ silent returns on validation failures. Namespace prefixes are present in some files (`[projectStore]`, `[historyStore]`) but absent from most. There is no severity control and no way to enable verbose output without editing source.

Active bugs are being reported that can't be located. The gaps below are why.

| Module / Function | Missing Behavior | Impact |
|---|---|---|
| `canvas-pixi/SelectionStateMachine.ts` ‚Äî `handleDown`, `handleMove`, `handleUp` | No logging on any state transition. Mode changes, drag start/end, and eraser operations are silent. | When selection stops working, there is no trace of which mode was active or which branch ran. **Critical.** |
| `canvas-pixi/SelectionStateMachine.ts` ‚Äî `eraseAtPoint` | Deletions have no log. No record of element ID, type, or position erased. | Accidental deletion reports can't be replayed. No audit trail for data loss. |
| `canvas-pixi/SelectionStateMachine.ts` ‚Äî `handleMove` snap | `snapPoint()` called on every move event with no logging of candidates, tolerance, or result. | Snap inconsistencies are completely blind. No way to tell whether guide lines failed to render or failed to compute. |
| `canvas-pixi/TerrainPaintHandler.ts` ‚Äî `onPointerDown`, `onPointerMove` | No log of which cells were painted, blocked-by-structure rejections, or brush size applied. | A paint stroke that partially applies or silently skips cells has no trace. |
| `canvas-pixi/PathDrawingHandler.ts` ‚Äî `finalizePath` | Silent returns if `pathTypeId` is missing, path type not found, or snapshot fails. The function returns without committing and without logging. | When a path doesn't save, no diagnostic output exists at the moment of failure. |
| `canvas-pixi/BoundaryHandler.ts` ‚Äî `doCommitBoundary` | `hasSelfIntersection` blocks the commit silently. No log of which edges intersected or how many vertices were in the shape. | Users get no feedback and there's no server-side or client-side record of the rejection. |
| `canvas-pixi/PlacementHandlers.ts` ‚Äî collision checks | `hasStructureCollision`, `hasSpacingCollision`, and `hasPlantStructureCollision` return `false` silently. | Silent placement failures have no indication of what collided. The `PlacementFeedback` work (ENG-61) targets the UI side, but the rejection reason still isn't logged. |
| `canvas-pixi/InteractionManager.ts` ‚Äî `onPointerDown` dispatch | Tool dispatch switch has no logging. Boundary drag state transitions and boundary-placing branch are silent. | Tool non-response has no trace. Can't tell which branch was entered or whether `isBoundaryPlacing()` returned the wrong value. |
| `snap/snapSystem.ts` ‚Äî `snapPoint`, `resolveAxis` | Candidate extraction, tolerance computation, and axis resolution produce no output. | Snap is a frequent source of bugs. Distinguishing a missed candidate from a wrong tolerance from a disabled snap requires either a debugger or adding logs manually. |
| `store/useHistoryStore.ts` ‚Äî `undo`, `redo` | State before/after undo is unlogged. `loadHistory` validation failures log the error but not which entries were rejected. | When undo stacks diverge between sessions, there's nothing to trace. |
| `store/useProjectStore.ts` ‚Äî `updateProject` | No logging of what action triggered the update. 15+ call sites all produce the same silent mutation. | Can't distinguish which handler corrupted state when a bug is reported. Adding an `actionName` parameter here would give all 15 sites free traceability. |
| `canvas-pixi/RenderScheduler.ts` ‚Äî `render` | `perfLogging` is off by default and requires a source edit to enable. Frame drops and rAF health are invisible. | Performance regressions in the canvas require code changes to diagnose, not a browser console flag. |
| `api/generateClient.ts` ‚Äî `sendGenerateRequest` | Request start, response status, and response timing are unlogged. Errors are caught and rethrown but the request body shape is never logged. | Generate failures produce a toast but no structured record of what was sent, what status came back, or how long the request took. |
| `store/useGenerateStore.ts` ‚Äî `generate` | No log on enter, no log on success, no log on abort vs timeout. Module-level `isTimeoutAbort` flag state is invisible. | Timeout vs user-cancel vs network error are distinguishable in `mapErrorToToast` but not traceable after the fact. |

**What the codebase needs.** A lightweight structured logger, either `loglevel` or a thin wrapper around `console`, with per-module namespaces and a `localStorage` flag for enabling verbose output in production. The pattern in `WelcomeScreen.tsx` (namespaced, structured fields, explicit debug vs error levels) is the right shape. It's the only file that does this consistently.

Priority order for retrofitting:

1. `useProjectStore.updateProject` ‚Äî add an `actionName: string` param. All 15+ callers get traceability for free.
2. `SelectionStateMachine` ‚Äî log mode transitions at debug level with element IDs and coordinates.
3. Tool handlers (`PlacementHandlers`, `BoundaryHandler`, `TerrainPaintHandler`) ‚Äî log silent rejection paths at debug level with reason.
4. `snapSystem.ts` ‚Äî log snap results at debug level (gated on a flag to avoid noise on every pointermove).
5. `useGenerateStore.generate` ‚Äî log request start with sanitized body shape, response status, and duration.
